## 정규화
- 데이터 중복을 최소화하고, 데이터의 무결성과 일관성을 보장하기 위해 테이블을 구조화

### 정규화 목적
1. 데이터 중복 최소화

2. 데이터 이상(anomaly) 제거
    - 삽입
    - 수정
    - 삭제 이상

3. 데이터 일관성 유지

4. 유지보수 및 확장성 향상

### 정규화 단계
| 정규형                               | 내용                                             | 예시 설명                                                              |
| --------------------------------- | ---------------------------------------------- | ------------------------------------------------------------------ |
| **1정규형 (1NF)**                    | 모든 속성은 **원자값(Atomic)** 만 가져야 함                 | 한 컬럼에 여러 값이 들어가면 안 됨 → 예: `phone = "010-1234-5678, 02-111-2222"` ❌ |
| **2정규형 (2NF)**                    | **부분 함수 종속성 제거** (기본키의 일부분에 종속된 컬럼 제거)         | 복합키의 일부에만 의존하는 컬럼을 별도 테이블로 분리                                      |
| **3정규형 (3NF)**                    | **이행적 함수 종속성 제거** (비키 속성이 다른 비키 속성에 의존하지 않게 함) | 예: `학생(학번, 학과, 학과장)` → 학과장은 학과에 종속 → 학과 테이블로 분리                    |
| **BCNF (Boyce-Codd Normal Form)** | 모든 결정자가 후보키여야 함                                | 3NF보다 더 엄격한 형태                                                     |



## 비정규화
- 정규화된 테이블을 의도적으로 중복시켜, 조회 성능을 높이는 것

### 비정규화 목적
1. 조회 성능 향상
2. 복잡한 조인(Join) 최소화
3. 데이터 웨어하우스나 리포트성 조회에 유리

### 단점
1. 데이터 중복 증가
2. 데이터 불일치 가능성 (일관성 저하)
3. 업데이트/삭제 시 여러 곳을 수정해야 함


## ACID
| 항목                        | 설명                                      | 예시                                        |
| ------------------------- | --------------------------------------- | ----------------------------------------- |
| **A - Atomicity (원자성)**   | 트랜잭션의 모든 작업이 **전부 수행되거나 전혀 수행되지 않아야 함** | 송금: A 계좌 -10, B 계좌 +10 → 둘 다 성공 or 둘 다 실패 |
| **C - Consistency (일관성)** | 트랜잭션 수행 전후로 **DB 제약조건을 항상 만족**해야 함      | 잔액이 음수가 되면 안 됨 등 무결성 유지                   |
| **I - Isolation (고립성)**   | 동시에 수행되는 트랜잭션들이 **서로의 중간 결과를 볼 수 없음**   | 동시에 주문하더라도 각각의 트랜잭션은 독립적으로 수행             |
| **D - Durability (지속성)**  | 트랜잭션이 완료(commit)되면 **결과가 영구히 저장**되어야 함  | 시스템 다운돼도 커밋된 데이터는 보존                      |
