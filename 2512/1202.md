## 인터페이스와 추상 클래스의 차이에 대해 설명해 주세요.

| 특징 | 인터페이스 (Interface) | 추상 클래스 (Abstract Class) |
| :--- | :--- | :--- |
| **목적** | **기능 정의 (What)** : 클래스가 무엇을 할 수 있는지 정의하는 데 중점 ; 관련된 기능(행위)을 묶음. | **유형 계층 구조 및 공통 구현 (How)** : 밀접하게 관련된 클래스 간의 공통 특성 및 기본 구현을 제공하는 데 중점. |
| **포함 가능 요소** | 상수(public static final) ; 추상 메서드(public abstract) ; (Java 8 이후) default 및 static 메서드. | 필드(변수) ; 추상 메서드 ; 구체(완전) 메서드 ; 생성자. |
| **구현 유무** | (대부분의 경우) **구현이 없음** ; **정의만 제공** ; (Java의 default 메서드는 예외). | **부분적인 구현 포함 가능** ; 일부 메서드는 구현을 제공하고, 일부는 추상 메서드로 남겨둘 수 있음. |
| **상속/구현** | 클래스는 **여러 개**의 인터페이스를 **구현(implements)** 할 수 있음. | 클래스는 **하나**의 추상 클래스 또는 일반 클래스를 **상속(extends)** 할 수 있음. |
| **변수 (필드)** | **상수(Constant)** 만 허용 (public static final) ; 객체 상태를 가질 수 없음. | 일반 변수(인스턴스 변수)를 가질 수 있음 ; 객체의 상태(State)를 정의할 수 있음. |
| **접근 지정자** | 모든 메서드는 기본적으로 **public** (명시하지 않아도 public). | abstract가 아닌 메서드와 필드는 어떤 접근 지정자(public, protected, private 등)도 가질 수 있음. |
| **예시** | `Serializable`, `Comparable`, `Runnable` (구현에 상관없이 특정 능력을 가짐). | `AbstractMap`, `AbstractList` (공통된 속성과 기본 동작을 공유하는 컬렉션). |


## 왜 클래스는 단일 상속만 가능한데, 인터페이스는 2개 이상 구현이 가능할까요?

- 객체지향 프로그래밍(OOP)에서 발생하는 **"다이아몬드 문제(Diamond Problem)"**를 피하고 코드의 모호성을 방지

### 클래스의 단일 상속만 허용하는 이유
- 클래스는 **구현(Implementation)**을 상속
- 다이아몬드 구조 방지
    1. Class A에 methodX()가 정의
    2. Class B와 Class C가 모두 Class A를 상속하고, 각자 methodX()를 다른 방식으로 오버라이딩(Overriding)
    3. Class D가 Class B와 Class C 모두를 상속
    4. Class D에서 methodX()를 호출하면, 컴파일러는 B의 구현을 사용해야 할지 C의 구현을 사용해야 할지 결정할 수 없어 모호성(Ambiguity)이 발생

### 인터페이스의 다중 구현이 가능한 이유
- **계약(Contract)**만을 정의하며, 본래 구현(Implementation)을 포함하지 않음
- 구현의 부재
    - 인터페이스는 클래스의 methodX()와 같은 구체적인 코드를 상속하지 않고, 단순히 methodX()가 존재해야 한다는 '선언'만을 강제
- 모호성 없음
    - Class D가 Interface B와 Interface C를 동시에 구현할 때, 두 인터페이스 모두 methodX()를 선언하더라도, Class D는 자신이 직접 methodX()의 유일한 구현 코드를 제공
