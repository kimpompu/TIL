## 상속(Inheritance)

### 개념 : 부모와 자식 관계 (IS-A)
**상속(Inheritance)** 은 기존 클래스(부모/슈퍼 클래스)의 속성과 기능을 새로운 클래스(자식/서브 클래스)가 물려받아 그대로 사용하거나 확장하는 기능입니다.

### 장점과 단점

| 구분 | 내용 |
| :--- | :--- |
| **장점** | **코드 재사용성** : 공통된 코드를 부모 클래스에서 관리하여 중복을 줄일 수 있습니다.<br>**확장성** : 기존 클래스를 수정하지 않고 기능을 확장하기 용이합니다. |
| **단점** | **강한 결합도** : 부모 클래스가 변경되면 자식 클래스도 영향을 받아 유지보수가 어려울 수 있습니다.<br>**유연성 저하** : 상속 구조가 깊어지면 복잡도가 증가하고, 불필요한 기능까지 상속받을 수 있습니다. |


### 객체지향 설계의 5가지 원칙
**SOLID** 는 유지보수하기 쉽고, 유연하며, 확장이 용이한 소프트웨어를 만들기 위한 5가지 설계 원칙의 앞 글자

1.  **SRP(Single Responsibility Principle, 단일 책임 원칙)** : 하나의 클래스는 하나의 책임만 가져야 합니다. 클래스를 변경하는 이유는 단 하나여야 합니다.
2.  **OCP(Open/Closed Principle, 개방-폐쇄 원칙)** : 확장에는 열려 있어야 하고, 변경에는 닫혀 있어야 합니다. 기존 코드를 수정하지 않고 기능을 추가할 수 있어야 합니다.
3.  **LSP(Liskov Substitution Principle, 리스코프 치환 원칙)** : 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 합니다. 부모 클래스의 인스턴스 대신 자식 클래스의 인스턴스를 사용해도 프로그램의 로직이 깨지지 않아야 합니다.
4.  **ISP(Interface Segregation Principle, 인터페이스 분리 원칙)** : 클라이언트는 자신이 사용하지 않는 메서드에 의존하지 않아야 합니다. 하나의 거대한 인터페이스보다는 구체적인 여러 개의 인터페이스가 낫습니다.
5.  **DIP(Dependency Inversion Principle, 의존 역전 원칙)** : 고수준 모듈은 저수준 모듈에 의존해서는 안 되며, 둘 다 추상화에 의존해야 합니다. 즉, 구체적인 클래스가 아닌 인터페이스나 추상 클래스에 의존해야 합니다.

## 인터페이스(Interface)와 추상 클래스(Abstract Class)의 차이점

### 역할과 구현의 차이

| 특징 | 추상 클래스(Abstract Class) | 인터페이스(Interface) |
| :--- | :--- | :--- |
| **목적** | **상속 및 확장 (IS-A)** : 관련된 클래스들 간의 공통 코드를 재사용하고 확장하는 데 초점. | **행위 규약 (CAN-DO)** : 서로 다른 클래스들이 동일한 동작을 수행하도록 강제하는 데 초점. |
| **다중 상속** | 불가능 (단일 상속만 지원하는 언어가 많음, 예 : Java) | **가능** (여러 인터페이스를 동시에 구현 가능) |
| **구성 요소** | 추상 메서드뿐만 아니라 일반 메서드, 멤버 변수(상태), 생성자 등을 가질 수 있음. | 주로 추상 메서드와 상수(static final)만 가질 수 있음. (Java 8부터 default 메서드 가능) |
