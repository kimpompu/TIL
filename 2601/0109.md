## HTTP와 HTTPS의 차이점을 설명해 주세요.

* **HTTP** : 데이터를 평문(Text)으로 전송하기 때문에 네트워크 상에서 패킷을 가로채면 내용을 누구나 볼 수 있어 보안에 취약함. 
* **HTTPS** : HTTP에 **SSL/TLS** 프로토콜을 추가하여 데이터를 **암호화**해서 전송함. 이로 인해 도청과 변조를 방지할 수 있으며, 검색 엔진 최적화(SEO)에서도 HTTPS 사용이 권장됨.

## TCP와 UDP의 차이점을 설명해주세요.

* **TCP(Transmission Control Protocol)** : 연결 지향적이며 3-way Handshake 과정을 통해 신뢰성 있는 데이터 전송을 보장함. 데이터의 순서를 보장하고 손실 시 재전송을 수행함.
* **UDP(User Datagram Protocol)** : 비연결형 프로토콜로, 연결 설정 없이 데이터를 전송하며 신뢰성보다는 **속도**와 연속성이 중요한 스트리밍이나 실시간 게임 서비스에 주로 사용됨.

| 구분 | TCP | UDP |
| :--- | :--- | :--- |
| **연결 방식** | 연결 지향 (Connection-oriented) | 비연결형 (Connectionless) |
| **신뢰성** | 높음 (순서 보장, 재전송) | 낮음 (데이터 유실 가능) |
| **속도** | 상대적으로 느림 | 빠름 |
| **사용 사례** | 웹(HTTP), 이메일, 파일 전송 | 실시간 스트리밍, VoIP, 온라인 게임 |

## 서버 사이드 렌더링과 클라이언트 사이드 렌더링의 차이점을 설명해주세요.

* **SSR(Server Side Rendering)** : 서버에서 HTML을 완성하여 브라우저로 보내주는 방식임. 초기 로딩 속도가 빠르고 검색 엔진 크롤러가 내용을 읽기 쉬워 **SEO**에 유리함. 하지만 페이지 이동 시마다 서버에 요청을 보내야 하므로 서버 부하가 클 수 있음.
* **CSR(Client Side Rendering)** : 브라우저가 자바스크립트를 다운로드하여 클라이언트 측에서 동적으로 HTML을 생성하는 방식임. 초기 로딩은 느리지만 이후 인터랙션이 빠르고 서버 부하가 적음.



## 웹소켓(WebSocket)이란 무엇이며, 실시간 통신에서 이를 어떻게 사용할 수 있는지 설명해주세요.
* **웹소켓** : 하나의 TCP 연결을 통해 클라이언트와 서버가 서로 자유롭게 데이터를 주고받을 수 있는 **전이중(Full-duplex)** 통신 프로토콜. 일반적인 HTTP 통신은 클라이언트의 요청이 있어야만 서버가 응답할 수 있지만, 웹소켓은 서버가 클라이언트에게 능동적으로 데이터를 보낼 수 있음. 따라서 실시간 채팅, 주식 시세 업데이트, 멀티플레이 게임 등 실시간성이 중요한 서비스에서 사용

## 웹훅(Webhook)이란 무엇인지 설명해주세요.

### 역방향 API(Reverse API)
* **웹훅** : 서버에서 특정 이벤트가 발생했을 때, 클라이언트가 지정한 URL로 실시간으로 데이터를 보내주는(Push) 기능. 클라이언트가 서버에 계속 데이터를 물어보는 폴링(Polling) 방식과 달리, 사건이 터졌을 때 서버가 알려주는 방식이므로 리소스 낭비가 적고 실시간성이 높음 
ex) 결제 완료 시 결제 서버가 쇼핑몰 서버로 알림 전송

## IP 주소와 DNS(Domain Name System)의 관계를 설명해 주세요.

* **IP 주소** : 컴퓨터 네트워크에서 장치들이 서로를 인식하고 통신하기 위한 숫자로 된 실제 주소. 문자 형태의 도메인 이름을 사용함. **DNS**는 전화번호부처럼 도메인 이름을 입력받아 그에 해당하는 실제 IP 주소를 찾아주는 시스템.



## NAT(Network Address Translation)이란 무엇이며, 이를 사용하는 이유를 설명해 주세요.

* **NAT** : 라우터 등을 통해 패킷이 외부 네트워크로 나갈 때, 내부의 사설 IP 주소를 공인 IP 주소로 변환해 주는 기술
* 사용이유
    1.  **IPv4 주소 절약** : 하나의 공인 IP로 여러 대의 내부 기기가 인터넷을 사용할 수 있게 하여 IP 고갈 문제를 완화
    2.  **보안 강화** : 외부에서는 내부 네트워크의 사설 IP를 알 수 없으므로, 내부망을 보호하는 방화벽 역할을 수행



## HTTP Keep-Alive의 역할과 성능에 미치는 영향을 설명해 주세요.

### 연결 재사용
HTTP는 기본적으로 비연결성(Connectionless)이라 요청마다 TCP 연결을 맺고 끊는 3-way Handshake 비용이 발생함. **Keep-Alive**는 이미 맺어진 TCP 연결을 끊지 않고 일정 시간 동안 유지하여, 후속 요청들이 해당 연결을 **재사용**할 수 있게 해줌. 이를 통해 핸드쉐이크 오버헤드를 줄여 응답 속도를 개선하고 서버의 CPU 부담을 줄임.

## TLS(Transport Layer Security)와 SSL(Secure Sockets Layer)의 차이점을 설명해 주세요.

### 표준화와 버전
**SSL**은 넷스케이프사에서 개발한 암호화 프로토콜이며, **TLS**는 SSL을 기반으로 국제 표준 기구(IETF)에서 표준화하여 만든 후속 버전임. 기술적으로 TLS가 더 강력한 암호화 알고리즘을 지원하며 보안성이 높음. 현재 SSL의 모든 버전은 보안 취약점으로 인해 사용이 중단되었으며, 우리가 흔히 SSL 인증서라고 부르는 것은 실제로는 TLS 프로토콜을 사용함.

## 서버 부하 분산(Load Balancing) 방식 중 라운드 로빈(Round Robin) 방식과 IP 해시 방식의 차이점을 설명해 주세요.

### 분배 로직의 차이



* **라운드 로빈(Round Robin)** : 서버 리스트를 순서대로 돌아가며 요청을 하나씩 배분하는 방식임. 구현이 간단하고 서버 스펙이 동일할 때 유용하지만, 세션 유지(Sticky Session)가 보장되지 않음.
* **IP 해시(IP Hash)** : 클라이언트의 IP 주소를 해싱하여 특정 서버에 매핑하는 방식임. 사용자가 항상 같은 서버로 연결되므로 세션 유지가 필요한 서비스에 적합하지만, 특정 서버에 트래픽이 몰릴 수 있음.
