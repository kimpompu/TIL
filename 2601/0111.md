## 단위 테스트(Unit Test)와 통합 테스트(Integration Test)의 차이점을 설명해 주세요.

### 정의 및 범위
* **단위 테스트(Unit Test):** 응용 프로그램에서 테스트 가능한 가장 작은 단위(일반적으로 메서드나 클래스)를 격리하여 독립적으로 검증하는 테스트입니다. 외부 의존성(데이터베이스, 네트워크 등)은 Mock 객체로 대체하여 배제합니다.
* **통합 테스트(Integration Test):** 단위 테스트를 통과한 모듈들을 결합하여 모듈 간의 상호작용 및 데이터 흐름이 올바르게 작동하는지 확인하는 테스트입니다. 실제 데이터베이스나 외부 API와의 연결을 포함하는 경우가 많습니다.

### 주요 차이점
| 구분 | 단위 테스트 | 통합 테스트 |
| :--- | :--- | :--- |
| **대상** | 개별 함수, 메서드, 클래스 | 모듈 간 인터페이스, 시스템 전체 |
| **속도** | 매우 빠름 | 상대적으로 느림 |
| **목적** | 개별 로직의 논리적 결함 발견 | 모듈 간 상호작용 및 설정 오류 발견 |
| **복잡도** | 낮음 (격리된 환경) | 높음 (의존성 관리 필요) |

## TDD(Test Driven Development)가 무엇이며, 이를 적용한 경험이 있나요?

### 정의 및 프로세스
TDD는 코드를 작성하기 전 테스트 케이스를 먼저 작성하고, 이를 통과하기 위한 코드를 구현하는 개발 방법론입니다. Kent Beck에 의해 대중화되었으며, 'Red-Green-Refactor' 사이클을 반복합니다.
1. **Red:** 실패하는 테스트 케이스를 먼저 작성합니다.
2. **Green:** 테스트를 통과하기 위한 최소한의 코드를 작성합니다.
3. **Refactor:** 코드의 기능은 유지하면서 구조를 개선하고 중복을 제거합니다.

### 적용 효과
* **객관적 사실:** 코드의 테스트 커버리지가 자연스럽게 높아지며, 초기 설계의 결함을 조기에 발견할 수 있습니다. 
* **경험적 활용:** 요구사항이 명확한 비즈니스 로직(예: 복잡한 할인 정책 계산기) 구현 시 TDD를 적용하여 예외 상황에 대한 검증을 선제적으로 수행하고, 추후 리팩토링 시 안정성을 확보한 사례가 있습니다.

## 로깅(logging)이 중요한 이유와 효과적인 로깅 전략을 설명해 주세요.

### 로깅의 중요성
* **장애 추적:** 애플리케이션 실행 중 발생한 예외나 오류의 원인을 파악하는 유일한 사후 데이터입니다.
* **모니터링:** 시스템 상태, 트래픽 추이, 사용자 행동 패턴을 분석하는 근거가 됩니다.
* **감사(Audit):** 보안 사고 발생 시 접근 기록 및 데이터 변경 이력을 확인하는 용도로 사용됩니다.

### 효과적인 로깅 전략
1. **로그 레벨(Level) 구분:** `TRACE`, `DEBUG`, `INFO`, `WARN`, `ERROR`, `FATAL` 단계를 엄격히 구분하여 기록합니다.
2. **구조화된 로깅(Structured Logging):** 단순 텍스트가 아닌 JSON 등의 형식으로 기록하여 ELK Stack 등에서 검색 및 분석이 용이하게 합니다.
3. **의미 있는 컨텍스트 포함:** 타임스탬프, 요청 ID(Trace ID), 사용자 식별자, 에러 스택 트레이스 등을 포함해야 합니다.
4. **민감 정보 제외:** 개인정보(비밀번호, 주민번호 등)는 마스킹 처리하거나 로그에 남기지 않습니다.

## 서버에서 발생한 성능 문제를 디버깅한 경험이 있나요? 어떻게 해결했나요?

### 디버깅 절차 및 해결 방법
서버 성능 문제는 일반적으로 다음과 같은 단계로 해결합니다.
1. **지표 분석:** APM 도구(예: Pinpoint, New Relic)를 통해 응답 시간, 처리량, 에러율을 확인합니다.
2. **병목 지점(Bottleneck) 식별:** CPU 사용량 급증, 메모리 누수, I/O 대기 시간 중 원인을 파악합니다.
3. **프로파일링:** 스레드 덤프나 힙 덤프를 추출하여 데드락 여부나 과도한 객체 생성을 확인합니다.
4. **최적화 적용:** 불필요한 루프 제거, 캐시 적용, 커넥션 풀 크기 조절 등을 수행합니다.

## JUnit을 사용한 테스트 작성 경험이 있나요? 기본적인 사용법을 설명해 주세요.

### JUnit 개요
JUnit은 자바 프로그래밍 언어용 유닛 테스트 프레임워크입니다. 현재 JUnit 5(Jupiter)가 표준으로 사용됩니다.

### 주요 어노테이션 및 사용법
* **@Test:** 테스트 메서드임을 명시합니다.
* **@BeforeEach / @AfterEach:** 각 테스트 메서드 실행 전후에 실행될 로직을 정의합니다.
* **@BeforeAll / @AfterAll:** 전체 테스트 시작 전과 종료 후 한 번만 실행됩니다.
* **Assertions 클래스:** `assertEquals()`, `assertTrue()`, `assertThrows()` 등의 메서드로 결과를 검증합니다.

## 테스트 자동화를 구현하는 방법과 이를 통해 얻을 수 있는 이점에 대해 설명해주세요.

### 구현 방법
1. **프레임워크 선정:** JUnit(Back-end), Selenium/Cypress(Front-end/E2E) 등 적절한 도구를 선택합니다.
2. **CI/CD 파이프라인 통합:** Jenkins, GitHub Actions 등을 연동하여 코드가 Push될 때 자동으로 테스트가 수행되도록 구성합니다.
3. **테스트 데이터 관리:** 테스트 시마다 일관된 환경을 유지하기 위해 데이터베이스 초기화 스크립트나 In-memory DB를 활용합니다.

### 이점
* **회귀 테스트 효율성:** 코드 수정 시 기존 기능이 파괴되었는지 즉각 확인 가능합니다.
* **릴리스 속도 향상:** 수동 테스트 시간을 단축하여 지속적 배포(CD)가 가능해집니다.
* **정확성:** 휴먼 에러를 방지하고 반복적인 테스트 수행 시 일관성을 보장합니다.

## 성능 테스트(Performance Testing)와 부하 테스트(Load Testing)의 차이점을 설명해 주세요.

### 개념적 차이
* **성능 테스트 (Performance Testing):** 시스템의 속도, 반응성, 안정성을 평가하는 전체적인 범주의 테스트입니다.
* **부하 테스트 (Load Testing):** 성능 테스트의 일종으로, 예상되는 실제 사용자 부하 내에서 시스템이 어떻게 동작하는지 측정합니다.

### 상세 비교
| 구분 | 부하 테스트 (Load Testing) | 스트레스 테스트 (Stress Testing) |
| :--- | :--- | :--- |
| **목적** | 임계치 내에서의 정상 작동 확인 | 임계치를 넘었을 때 시스템의 붕괴 방식 확인 |
| **부하 수준** | 기대되는 최대 사용자 부하 | 시스템이 견딜 수 없는 극한의 부하 |
| **측정 지점** | 응답 시간, 자원 사용률 | 장애 발생 지점(Break Point) |

## 버그가 발생했을 때 이를 추적하는 방법과 해결 과정에 대해 설명해주세요.

### 버그 추적 및 해결 프로세스
1. **재현(Replication):** 버그가 발생하는 정확한 조건과 절차를 파악하여 동일한 현상을 구현합니다.
2. **격리(Isolation):** 로그 분석 및 디버깅 도구를 사용하여 버그가 발생하는 코드 영역을 좁힙니다.
3. **원인 분석(Root Cause Analysis):** 로직 오류, 데이터 문제, 환경 설정 오류 중 근본 원인을 식별합니다.
4. **수정 및 검증:** 코드를 수정하고 단위 테스트를 통해 해당 버그가 해결되었음을 확인합니다.
